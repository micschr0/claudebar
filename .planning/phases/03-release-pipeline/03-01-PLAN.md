triggered on v*.*.* tag push. The pipeline produces prebuilt binaries for all
four targets, archives each as claudebar-{version}-{target}.tar.gz, aggregates

Purpose: Enable one-command binary installation for Phase 4's install.sh — users
download a static binary rather than compiling from source.

Create .github/workflows/release.yml with the following exact structure. Use short action tags matching rust.yml convention (per D-02) — not SHA-pinned.

**Trigger (REL-01):**
```
on:
  push:
    tags:
      - 'v[0-9]+.*'
```

**Job 1 — create-release (per D-01: prevents race condition):**
- runs-on: ubuntu-latest
- permissions: contents: write
- steps:
  - uses: actions/checkout@v4
  - uses: taiki-e/create-gh-release-action@v1
    with:
      token: ${{ secrets.GITHUB_TOKEN }}
- Purpose: Creates the GitHub Release before any build job tries to upload assets. Without this, two build jobs finishing simultaneously could both try to create the release and conflict.

**Job 2 — build (matrix, 4 targets per REL-02, REL-06, REL-07):**
- needs: [create-release]
- strategy.matrix.include — four exact entries:
  - { target: x86_64-unknown-linux-musl, os: ubuntu-latest }
  - { target: aarch64-unknown-linux-musl, os: ubuntu-latest }
  - { target: x86_64-apple-darwin, os: macos-13 }
  - { target: aarch64-apple-darwin, os: macos-14 }
- runs-on: ${{ matrix.os }}
- permissions: contents: write (REL-08)
- steps:
  1. uses: actions/checkout@v4
  2. uses: houseabsolute/actions-rust-cross@v1 (per D-03)
     with:
       command: build
       target: ${{ matrix.target }}
       args: "--locked --release"
       strip: true
     Note: The `strip: true` here only affects native builds. Cross-compiled ARM Linux targets are stripped via `strip = true` in Cargo.toml [profile.release] which propagates into the cross-rs Docker container. Do NOT add a manual strip step.
  3. uses: taiki-e/upload-rust-binary-action@v1
     with:
       bin: claudebar
       target: ${{ matrix.target }}
       archive: claudebar-$tag-$target
       checksum: sha256
       include: README.md,LICENSE
       token: ${{ secrets.GITHUB_TOKEN }}
     Note: `$tag` and `$target` are taiki-e action template variables resolved from GITHUB_REF automatically.
     Note: README.md and LICENSE both exist at repo root — no path prefix needed.
     This action uploads the .tar.gz archive directly to the GitHub Release (per REL-03).
  4. uses: actions/upload-artifact@v4
     with:
       name: sha256-${{ matrix.target }}
       path: claudebar-*-${{ matrix.target }}.tar.gz.sha256
     Purpose: Passes the per-target checksum file to the release job for aggregation (per D-05).

**Job 3 — release (aggregates checksums, per REL-04, REL-05):**
- needs: [build]
- runs-on: ubuntu-latest
- permissions: contents: write (REL-08)
- steps:
  1. uses: actions/download-artifact@v4
     with:
       pattern: sha256-*
       merge-multiple: true
       path: checksums/
  2. run: cat checksums/*.sha256 | sort > SHA256SUMS.txt
     Note: The sort ensures deterministic ordering regardless of which build job finished first.
  3. uses: softprops/action-gh-release@v2 (per D-04 — NOT v3, requires Node 24)
     with:
       generate_release_notes: true
       files: SHA256SUMS.txt
     env:
       GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
     Purpose: Uploads SHA256SUMS.txt to the release and sets auto-generated release notes on the release body (REL-05).

**Critical: do NOT set permissions at workflow level** — scope contents: write to individual jobs only, minimizing blast radius per the security threat model (avoids overly-broad GITHUB_TOKEN permissions).

**Do NOT add `macos-latest`** anywhere in the matrix — it silently changed to ARM in 2024. Always use macos-13 (x86_64) and macos-14 (aarch64) explicitly.

**Do NOT add a separate `dtolnay/rust-toolchain` step** — `houseabsolute/actions-rust-cross@v1` installs and manages the Rust toolchain internally. Adding a separate toolchain step can conflict.

**Do NOT add a separate `Swatinem/rust-cache` step** — `actions-rust-cross@v1` includes built-in rust caching via `use-rust-cache: true` (default). Adding an explicit cache step would conflict.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tag push → workflow trigger | Only tag pushes matching v[0-9]+.* trigger the release — no PR trigger, no fork access |
| build job → GitHub Release | GITHUB_TOKEN with contents: write scoped per-job; not workflow-level |
| action references → supply chain | Short tags (@v1, @v2) are consistent with rust.yml convention (D-02) — inherently less supply-chain-secure than SHA pins but accepted per project decision |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | action references (short tags per D-02) | accept | D-02 locks short tags to match rust.yml convention; project accepts this tradeoff; mitigate by using well-known publishers only (all approved in package legitimacy audit) |
| T-03-02 | Elevation of Privilege | GITHUB_TOKEN permissions | mitigate | scope `permissions: contents: write` to individual jobs (build, release) only — not at workflow level; create-release job also scoped |
| T-03-03 | Spoofing | workflow trigger (fork PR could run release) | mitigate | trigger is `on: push: tags:` only — not `pull_request`; fork PRs cannot push tags to upstream |
| T-03-04 | Tampering | archive naming via tag injection | accept | binary name hardcoded as `claudebar`; tag comes from `GITHUB_REF` (protected GitHub value, not user-controlled input) |
| T-03-05 | Information Disclosure | checksum algorithm (MD5/SHA1) | mitigate | `checksum: sha256` in taiki-e action — SHA256 only, no weak hashes |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | no npm/pip/cargo installs in this workflow — GitHub Actions only; all actions vetted in package legitimacy audit |
</threat_model>
